## 1. 检查 ELF 是 32 位还是 64 位

用 `readelf -h` 看 ELF Header：

```bash
readelf -h python
```

输出里 `Magic` 一行的第 5 个字节：

- `02` → 64 位
- `01` → 32 位

## 2. 强制编译 32 位程序

脚本里设置 `USE_32BITS=1`，并提前安装好 `gcc-X-multilib` / `g++-X-multilib`：

```cmake
if(USE_32BITS)
    message(STATUS "using 32bits")
    set(CMAKE_C_FLAGS   "${CMAKE_C_FLAGS}   -m32")
    set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -m32")
else()
endif(USE_32BITS)
```

## 3. C++ 编译流程

整体顺序：**预处理 → 编译 → 汇编 → 链接**。

![image](https://user-images.githubusercontent.com/8308226/235432846-a8548168-815f-4888-ab28-44f55dcf1813.png)

具体路径：

>源代码（source code）→ 预处理器（processor）→ 编译器（compiler）→ 汇编程序（assembler）→ 目标程序（object code）→ 链接器（Linker）→ 可执行程序（executables）

### 3.1 预处理

预处理相当于根据预处理指令组装新的 C++ 程序。结果是一个**没有宏定义、没有条件编译指令、没有特殊符号**的文件，语义和原文件相同：

- 处理所有以 `#` 开头的伪指令：删除 `#define` 并展开宏；处理 `#if` / `#ifdef` / `#elif` / `#else` / `#endif` 等条件编译；将 `#include` 包含进来（可递归）
- 删除所有注释
- 添加行号和文件名标识（编译时报错才能显示行号）
- 保留所有 `#pragma` 编译器指令
  - 例如 `#pragma once`：放在头文件最开始，保证只编译一次
  - 参考：<https://baike.baidu.com/item/%23pragma>

### 3.2 编译

将预处理后的文件做**词法 → 语法 → 语义**分析，确认指令合法后翻译成汇编代码。这一阶段会做检查和优化，并报告语法 / 重载 / 类型等错误：

- **词法分析**：从左到右一字一字扫描，按构词规则识别单词
- **语法分析**：把单词序列组合成「程序 / 语句 / 表达式」等语法短语
- **语义分析**：对结构正确的源程序做上下文相关检查，比如类型检查；典型错误如「在表达式中使用一个数组变量」「赋值语句左右类型不匹配」

### 3.3 汇编

将汇编代码翻译成机器指令，生成可重定位目标文件 `.o`（二进制）。
汇编几乎是一一对应的过程：每条汇编语句对应一条机器指令，没有复杂语法 / 语义，也不做指令优化，只查表翻译。

### 3.4 链接

链接器利用编译器产生的目标文件生成最终的可执行文件，把未定义符号引用替换成正确地址，最后产出动态链接库或可执行文件。
没链接相当于一页没有页码的目录——没法用。

## 4. 静态库和动态库

从编译角度看，最大的差别在于「是否发生过链接」。

- **静态库**：本质是 `.o` 文件的集合，**没有执行链接**。引用其他文件的符号也没解引用。这些其他文件可能是同一代码库下别的 `.o`，也可能是其他模块的静态库 / 动态库。
  - 静态库**按需链接**，粒度是 `.o`，而不是 `.a`：只有被引用到的符号所在 `.o` 会被写入应用程序。代码里没用到，链接器就不会真去链。
- **动态库**：目标文件经过链接处理就成了动态库。从操作系统角度看它包含指令和数据，结构上更接近应用程序；给动态库加上入口函数，它甚至可以像应用程序一样独立运行。

## 5. 编译裁剪

线上常见的两个动作：

- 区分 `debug` / `release`，线上裁掉调试信息
- 移除不需要的 lib / test 二进制，提高部署速度

控制对外导出符号的示例（`.def` 文件风格）：

```text
LIBRARY micontinuity_sdk

EXPORTS

GetErrMsg;
extern "C++" {
    "class::Get()";
    class::Packet::*;
};
```

## 7. 常见代码优化技术

通用的几类做法：

1. **公共子表达式消除（CSE）**：表达式 `e` 已计算过，期间相关变量没变，则后续出现可视为公共子表达式直接复用。
2. **删除无用代码**：永远到不了或没意义的代码会被清除。
3. **常量传播**：能在编译期算出的变量值直接替换为常量。
4. **常量折叠**：多个变量组成的可静态求值表达式直接替换为常量。
5. **复写传播**：值相同的变量可以互相替代，减少冗余。
6. **数组范围检查消除**：能静态推断变量取值范围时，循环里的边界检查可以消掉。
7. **方法内联**：把被调函数代码「拷」进调用点，省掉函数调用开销。
8. **逃逸分析**：分析对象动态作用域，若确认不发生方法逃逸 / 线程逃逸，可做：
   - **栈上分配**：把堆分配换成栈分配
   - **同步消除**：单线程访问时去掉同步
   - **标量替换**：对象的字段拆开放进寄存器，不必整体放在内存

### 7.1 寄存器分配

寄存器是位于 CPU / GPU 内部的少量高速存储，用于保存机器指令的操作数。寄存器位数与指令位宽一致，例如 128 位指令意味着每条指令最多处理 128 位数据，这是「向量化计算」的硬件基础。

**寄存器分配**就是把高级语言里几乎无限多的变量，映射到 CPU / GPU 上数量有限的物理寄存器；它可以工作在表达式 / 基本块 / 函数（全局）/ 整个程序级别。

参考：<https://zhuanlan.zhihu.com/p/55287942>

### 7.2 指令调度

指令调度用于提高指令级并行，从而提升 CPU 流水线性能。在不改变语义的情况下做两件事：

- 通过重排指令顺序，避免流水线停顿
- 避免非法或语义模糊的操作（涉及流水线时序问题或非互锁资源）

## 8. GCC 与基本优化选项

### 8.1 GCC 简介

GCC 全称是 GNU Compiler Collection，最早作为 C 编译器（GNU C Compiler）开发，现已支持 C / C++ / Java / Pascal 等语言：

- 可移植：支持多种硬件平台（ARM / X86 等）
- 不止本地编译，也能跨平台交叉编译（例如在 x86 上编译然后跑在 ARM 上）
- 多语言前端，模块化设计，方便扩展新语言、新 CPU 架构
- 自由软件

### 8.2 优化等级与展开

GCC 提供近百种优化选项，用来在 `{ 编译时间, 目标文件大小, 执行效率 }` 三者间做不同取舍。手段大致包括：精简指令、配合 CPU 流水、根据预测重排执行顺序、充分利用寄存器、展开简单调用 等。

#### `-O0`

不做任何优化，默认值。

#### `-O` / `-O1`

对程序做部分编译优化。对大函数会占用稍多编译时间和内存，但会尝试减小目标代码体积、缩短执行时间，**不执行**那些消耗大量编译时间的优化。代表性子项：

- `-fdefer-pop`：延迟栈弹出，多次函数调用一次性弹栈
- `-fmerge-constants`：跨编译单元合并相同常量（字符串、浮点常量）
- `-fthread-jumps`：跳转目的地有另一处条件比较时，按结果重定向
- `-floop-optimize`：执行循环优化，把常量表达式移出循环、简化退出条件、可选做强度削减或循环展开
- `-fif-conversion` / `-fif-conversion2`：尝试把条件跳转转换为等价的无分支形式（条件移动 / min / max / 标志 / abs / 算术技巧等）
- `-fdelayed-branch`：根据指令周期重排，尽量把指令前移到条件分支前以利用治理缓存
- `-fguess-branch-probability`：没有 profiling 反馈或 `__builtin_expect` 时，按随机模式猜分支概率
- `-fcprop-registers`：寄存器分配后再扫一遍，减少调度依赖、删除多余寄存器复制

#### `-O2`

比 `-O1` 更多优化，几乎包含「不引入时空折中」的所有优化项。代表性子项：

- `-fforce-mem`：算术前先把内存数据搬进寄存器
- `-foptimize-sibling-calls`：优化兄弟调用与尾递归调用
- `-fstrength-reduce`：循环强度削减并删除迭代变量
- `-fcse-follow-jumps` / `-fcse-skip-blocks`：CSE 时跨跳转 / 跨基本块继续扫描
- `-frerun-cse-after-loop` / `-frerun-loop-opt`：循环优化完成后再跑一次 CSE / 循环优化
- `-fgcse` / `-fgcse-lm` / `-fgcse-sm` / `-fgcse-las`：全局 CSE，及其与 load / store 提升 / 沉降相关的子优化
- `-fdelete-null-pointer-checks`：基于全局数据流分析删掉无用的空指针检查
- `-fexpensive-optimizations`：执行一些代价较高的优化
- `-fregmove`：尝试为 move 等简单指令重新分配寄存器，对双操作数机器尤其有用
- `-fschedule-insns` / `-fschedule-insns2`：指令重排，前者一次，后者在寄存器分配后再来一次
- `-fsched-interblock`：跨基本块调度
- `-fsched-spec-load` / `-fsched-spec-load-dangerous`：允许 load 指令做（更激进的）投机
- `-fcaller-saves`：通过保存 / 恢复 caller 寄存器使 callee 能用更多寄存器
- `-fpeephole2`：打开特定的 peephole 优化
- `-freorder-blocks` / `-freorder-functions`：重排基本块 / 函数，减少分支、提高代码局部性。后者依赖 `.text.hot` / `.text.unlikely` 段提示
- `-fstrict-aliasing`：强制严格的别名规则（不同类型变量不共享内存）
- `-funit-at-a-time`：在代码生成前先分析整个汇编代码，开启更多优化
- `-falign-functions` / `-falign-jumps` / `-falign-loops` / `-falign-labels`：函数 / 跳转 / 循环 / 标签的内存对齐
- `-fcrossjumping`：合并散在程序各处的相同代码

#### `-O3`

在 `-O2` 基础上再加：

- `-finline-functions`：内联简单函数，限制由编译器启发式决定（GCC 3.4.6 中默认上限 600，可通过 `-finline-limit=n` 调整）
- `-fweb`：构建用于保存变量的伪寄存器网络，配合 CSE / 循环优化
- `-frename-registers`：寄存器分配后利用剩余寄存器，去掉伪依赖
- `-funswitch-loops`：把循环里恒定的条件分支移出循环

### 8.3 一组冒泡排序耗时实测

| 优化等级 | 耗时    | 编译命令                       |
| :------ | :----- | :----------------------------- |
| 不优化  | 3039ms | `gcc sort.c -o sort`           |
| `-O1`   | 1323ms | `gcc -O1 sort.c -o sort.1`     |
| `-O2`   | 1129ms | `gcc -O2 sort.c -o sort.2`     |
| `-O3`   | 1126ms | `gcc -O3 sort.c -o sort.3`     |

可以看到 `-O2` 已经吃掉了主要收益，`-O3` 仅锦上添花。

## 9. perf

### 9.1 简介

Perf（Performance Event）是内置在 Linux 内核源码树中的性能剖析工具，基于事件采样原理，针对处理器和操作系统层面的性能指标做分析，常用于性能瓶颈定位、热点函数定位。

原理：每隔固定时间在每个 CPU 核上产生一个中断，中断里读取当前 pid / 函数符号并累计计数；运行越久的函数被击中的概率越大，从而推断出 CPU 时间分布。

![image](https://user-images.githubusercontent.com/8308226/235433399-9ffdc538-54ea-4175-a5dc-519090c9e849.png)

### 9.2 打开 CPU 时间监听权限

```bash
# 1. 立即生效（重启失效）
echo -1 > /proc/sys/kernel/perf_event_paranoid

# 2. 永久生效：写入 /etc/sysctl.conf
kernel.perf_event_paranoid = -1
```

## 10. FDO 与 AutoFDO

### 10.1 FDO 基本原理

FDO（Feedback-Directed Optimization）是 GCC 等编译器的特性：编译器很难判断分支是否会跳，OoO（Out-of-Order，预测执行）就是想缓解这个问题。FDO 的做法是先编译一个 **Instrumented 版本**（基于 gcov），运行一次后从 `.gcda` 文件里拿到真实跳转概率，再用这份数据指导第二次编译。

GCC 中的传统反馈式优化基于**插桩**收集边和值的性能信息，然后用这些信息指导：

- 指令调度
- 基本块重排
- 函数拆分
- 寄存器分配

### 10.2 FDO 步骤

FDO 与插桩高度耦合，GCC 要求两次编译使用相同的内联决策和优化选项，否则插桩前后的控制流图无法对齐。

```bash
# 1. 用 -fprofile-generate 生成插桩版本
gcc sort.c -o sort_instrumented -fprofile-generate

# 2. 运行插桩版本生成 .gcda
./sort_instrumented
# Bubble sorting array of 30000 elements
# 3622 ms

# 3. 基于 .gcda 重新编译并运行
gcc -O3 sort.c -o sort_fdo -fprofile-use=sort.gcda
./sort_fdo
# Bubble sorting array of 30000 elements
# 1161 ms
```

`-O3` 单独 1123ms，FDO 后 1161ms 略有波动，长期统计上 FDO 通常能再带来 3% 上下的额外收益。

> 实际开发里 FDO 用得不多：插桩版本拿到生产运行成本高、插桩本身有运行时开销、测试时也很难造出有代表性的负载。AutoFDO 就是为了解决这些问题。

### 10.3 AutoFDO 基本原理

AutoFDO 由 Google 提出，已集成到 GCC。和插桩式 FDO 不同，它**用 perf 采样**收集性能信息，然后用独立工具把 `perf.data` 转成 gcov 格式，再用这份数据指导优化。它跳过了插桩步骤，因此：

- 性能信息可以直接在生产环境收集
- 开发 / 测试期的数据也能拿来指导编译
- 适合时间关键型代码（如内核），插桩式 FDO 不适合内核而 AutoFDO 适合
- 当前基于插桩的 FDO 不支持获取内核执行计数信息

### 10.4 AutoFDO 步骤

#### 第一步：生成 perf 数据

AutoFDO 依赖处理器的 `BR_INST_RETIRED:TAKEN` 事件，事件名各平台不同。可以用 `ocperf`（pmu-tools 的一部分）来收集，省去查事件名的麻烦：

```bash
# 用 ocperf 采集
ocperf.py record -b -e br_inst_retired.near_taken:pp -- ./sort
# Bubble sorting array of 30000 elements
# 3731 ms
# [ perf record: Woken up 7 times to write data ]
# [ perf record: Captured and wrote 1.580 MB perf.data (3902 samples) ]

# 也可以直接用 perf
perf record -e br_inst_retired:near_taken -b -o perf.data -- your_program
```

把 `perf.data` 转成 gcov 格式：

```bash
create_gcov --binary=./sort --profile=perf.data --gcov=sort.gcov -gcov_version=1
# 注意：gcov_version 必须等于 1，这是 AutoFDO 当前唯一支持的版本
```

#### 第二步：用性能信息指导编译优化

GCC 会从 gcov 文件中读出：

- 函数名 / 文件名
- 源文件级性能信息（内联栈 → 采样计数的映射）
- 模块性能信息（模块 → 辅助模块映射）

```bash
gcc -O3 -fauto-profile=sort.gcov sort.c -o sort_autofdo
./sort_autofdo
# Bubble sorting array of 30000 elements
# 1160 ms
```

效果与传统 FDO 接近。

## 11. LTO（链接时优化）

LTO 在链接阶段做整体优化，能降低目标码体积，例如内核可减小 10% 以上、提速几个百分点；代价是需要更多内存和更长的编译时间。

核心思路：编译单文件时把编译器中间表示（GIMPLE）放进目标文件，链接阶段把所有 GIMPLE 加载进单一映像，重新做整体优化、生成最终代码。LTO 在 GCC 4.5 出现，到 4.7 才真正可用。

![image](https://user-images.githubusercontent.com/8308226/235433677-9c082345-956c-4e02-9e27-2d9d6b187001.png)

开启 LTO（Monolithic）后，`.o` 文件会附带优化信息，链接阶段先合成单一整体 `.o`，再与 framework 链接生成可执行文件：

![image](https://user-images.githubusercontent.com/8308226/235433714-35410be8-0b19-44d7-ae15-768e1320359e.png)

主要好处：

1. 函数内联化
2. 删除无用代码
3. 全局视角的优化

相对传统优化的优势：

1. 完成传统编译器无法做到的过程间优化（常量传播、生存期分析等）
2. 针对库函数在特定上下文做进一步优化（caller 处发现 inline 库函数更合适，就不再调用库函数）
3. LIR 层级优化：针对特定芯片指令集（cache 相关、指令对齐等）
4. 根据链接器最终确定的地址信息做优化（比如 L2 → L1 命中相关）

很多软件不开 LTO 的常见原因：

- 大型软件分布式编译，整体周期太长
- 钩子函数优化可能引入风险
- 调试体验受损

## 12. LLVM

### 12.1 基本原理

LLVM 是用 C++ 写的编译器构建框架，用于优化任意语言的 compile-time / link-time / run-time / idle-time。可以从两个角度看：

- 广义 LLVM：整个 LLVM 编译器架构，包括前端 / 后端 / 优化器，以及大量库与模块
- 狭义 LLVM：只指代码生成 / 优化的后端模块和库

传统编译器分三段：**前端 → 优化器 → 后端**：

- 前端做源码分析、语法检查、构建抽象语法树（AST）
- 中间转成统一表示
- 后端生成机器码

![image](https://user-images.githubusercontent.com/8308226/235433766-b79a9603-ee38-4834-beb9-1346a0e28516.png)

LLVM 也是这三段，但关键差异是：**对所有语言提供统一中间表示 IR**。

- 前端用各种工具把不同语言转成 LLVM IR
- 中间优化器只面向 IR，做一系列 pass
- 后端把优化后的 IR 翻译成目标平台机器码

这样不同语言可以共享同一套优化和后端。

![image](https://user-images.githubusercontent.com/8308226/235433797-05262777-917d-4bd6-9dec-e72e608ee2f9.png)

### 12.2 Clang

Clang 是 LLVM 的子项目，是基于 LLVM 架构的 C / C++ / Objective-C 编译器**前端**。相比 GCC：

- 编译速度快（在某些平台 Debug 模式下编译 OC 比 GCC 快约 3 倍）
- 占用内存小（生成的 AST 大约是 GCC 的 1/5）
- 模块化设计：基于库，便于 IDE 集成与重用
- 诊断信息可读性强：编译过程中保留大量 metadata，利于调试和报错
- 设计清晰简单，易于扩展

整体路径：

>源代码（C/C++）→ Clang → 中间代码（一系列 Pass 优化）→ 机器码

![image](https://user-images.githubusercontent.com/8308226/235433821-526ec144-730c-4d68-98b3-9a2c31353844.png)

## 13. BOLT 编译后优化

完整编译流程里，采样优化可以发生在多个阶段：AutoFDO 在编译阶段、LTO 在链接阶段、Ispike 在后链接阶段（post-link）。

AutoFDO 一个主要问题是把采样数据映射回中间文件比较难；后链接阶段（post-link）由于在编译 / 链接优化之后才动手，精度会更高。BOLT 就是在这一思路下产生的**静态二进制优化器**，主要优化代码 layout，与 AutoFDO / LTO 等是互补关系。

BOLT 由 Facebook 提出：通过 perf 采集运行数据，对程序符号重新排列，提升 CPU 指令 cache 命中率。

### 13.1 收集 perf 数据

```bash
./perf record -F 4000 -e cycles:u -o perf.data -j any,u -p $pid -- sleep $sleep
# 参数说明：
# -F        采样频率
# -e        性能事件，cycles:u 表示用户态 cycles
# -o        输出文件，默认 perf.data
# -p        被采集进程的 pid
# -- sleep  采样时长
```

### 13.2 转换成 BOLT 格式

```bash
perf2bolt -p perf.data -o perf.fdata <executable>
# perf2bolt：把 perf 数据转成 bolt fdata 文件
# -p：原始 perf 文件
# -o：转换后的目标 fdata
```

### 13.3 用 fdata 生成优化版二进制

```bash
llvm-bolt bin -o bin.bolt -data=bin.fdata \
  -align-macro-fusion=all \
  -reorder-blocks=cache+ \
  -reorder-functions=hfsort+ \
  -split-functions=3 -split-all-cold -split-eh \
  -dyno-stats \
  -icf=1 \
  -update-debug-sections
# bin       原始二进制
# bin.bolt  优化后的二进制
# -data     由 perf2bolt 转出的 fdata
```

### 13.4 多模式合并

如果目标程序有多个运行模式，可以分别采集再合并：

```bash
merge-fdata *.fdata > combined.fdata
# merge-fdata：把多个 fdata 合并成一个
```

## 14. configure / cmake 小技巧

### 14.1 configure 跨平台编译

`--host` 用编译工具链，`prefix` 用 `bin/` 下工具链的前缀部分（例如 `arm-fsl-linux-gnueabi-gcc` 的前缀）：

```bash
CC=path/arm-linux-gcc ./configure \
    --cache-file=cache_file_0 \
    --prefix=/home/eatjpg/arm-bin \
    --host=arm-fsl-linux-gnueabi
```

### 14.2 cmake：release 自动 strip

```cmake
function(utils_strip TARGET)
    add_custom_command(
        TARGET "${TARGET}" POST_BUILD
        DEPENDS "${TARGET}"
        COMMAND $<$<CONFIG:release>:${CMAKE_STRIP}>
        ARGS --strip-all $<TARGET_FILE:${TARGET}>
    )
endfunction()

# 用法：
utils_strip(${mbedcrypto_target})
```

### 14.3 release 版 strip 行为差异

- 可执行文件（`.exe` / 无后缀）一般 release 模式会自动 strip
- so 通常需要额外脚本 strip

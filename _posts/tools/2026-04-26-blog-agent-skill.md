> 本文把之前零散记录的 Agent 使用经验，整理成一份可执行的 Blog Agent Skill。后续对本站进行文章发布、内容调整、结构升级或样式优化时，优先按本文流程执行。

## 1. Skill 目标

这个 Skill 不是某个单一工具的配置，而是一套面向博客维护的工作方式：

- 把用户的原始想法整理成可发布、可维护的博客内容；
- 保持本站已有的 Jekyll 文章结构、Front Matter、标签与分类风格一致；
- 修改前先理解上下文，修改后做必要验证；
- 对不确定信息显式标注，不编造事实、命令或结论；
- 每次变更尽量小步提交，避免顺手改动无关内容。

## 2. 适用场景

适用于以下 blog 工作：

- 新增文章：把笔记、提纲、命令记录、排查过程整理为 `_posts/<category>/YYYY-MM-DD-title.md`；
- 批量专题：先把同一专题的原始资料放入 `_patch/`，由 Agent 分批整理到可发布状态；
- 调整文章：修正错别字、补充结构、重排章节、统一术语；
- 升级专题：把多篇散乱文章合并为系列、索引、FAQ 或 checklist；
- 站点维护：调整配置、样式、构建脚本、目录分类或标签；
- Agent 资料沉淀：把对话中的经验总结为可复用的流程文档。

不适用于：

- 未经确认的大规模迁移；
- 删除历史文章；
- 引入新的构建框架或发布系统；
- 把未经验证的私密信息、密钥、账号或内部地址写进文章。

## 3. 标准工作流

### 3.1 识别任务类型

收到需求后，先判断属于哪一类：

1. **解释类**：只回答问题，不改文件；
2. **计划类**：给出计划，不创建计划文件；
3. **内容类**：新增或整理博客文章；
4. **工程类**：修改站点配置、样式、脚本或构建流程。

只有内容类和工程类需要改仓库文件。

### 3.2 修改前理解上下文

动手前至少确认：

- 目标内容应该放在哪个分类目录；
- 是否已有相近文章，避免重复；
- 现有文章的 Front Matter、标题、标签风格；
- 修改是否会影响构建、样式或导航；
- 用户是否要求创建 PR、草稿或正式发布文章。

### 3.3 内容整理规则

整理原始笔记时按以下优先级处理：

1. 先保留核心结论；
2. 再补齐背景、适用场景和边界；
3. 命令、配置、错误日志独立成代码块；
4. 风险、限制、注意事项单独列出；
5. 最后补充 checklist、FAQ 或后续 TODO。

如果原始材料混杂多个主题，应拆成清晰章节；如果主题跨度太大，应建议拆分为多篇文章。

### 3.4 `_patch` 批量专题处理流程

当一次性上传同一类型、同一专题或同一批来源的原始笔记时，不要直接放入 `_posts/` 发布。推荐先放入 `_patch/` 作为暂存区，再由 Agent 按批次整理。

推荐目录结构：

```text
_patch/
  topic-name/
    00-index.md
    raw-note-1.md
    raw-note-2.md
```

处理流程：

1. **暂存原始资料**：用户或 Agent 把未整理材料放入 `_patch/<topic>/`，保持原始信息完整；
2. **建立处理清单**：在 `_patch/<topic>/00-index.md` 记录主题范围、资料列表、目标分类、预期输出文章；
3. **分批整理**：Agent 逐篇阅读原始资料，合并重复内容，拆分过长主题，补齐背景、步骤、风险和 FAQ；
4. **形成待发布稿**：整理后的文章先在 `_patch/<topic>/` 内完成校对，确认标题、标签、Front Matter 和目标路径；
5. **达到发布标准**：满足本文「完成标准」后，再移动到 `_posts/<category>/YYYY-MM-DD-title.md`；
6. **清理暂存区**：发布后删除或归档已处理原始材料，只保留仍需处理的内容。

验收要求：

- `_patch/` 中的内容默认视为未发布、未完成、可继续修改；
- `_patch/` 不作为正式博客内容入口，不要求每个文件都有 Front Matter；
- 移入 `_posts/` 前必须补齐 Front Matter、分类、标签、标题层级和安全检查；
- 同一专题拆成多篇文章时，应考虑增加索引、FAQ 或 checklist；
- 原始资料中如包含敏感信息，必须先脱敏，再进入可发布稿。

### 3.5 修改后验证

根据变更范围选择验证方式：

- 只改 Markdown：检查 Front Matter、标题层级、链接、代码块和中文标点；
- 改样式或脚本：运行现有构建命令；
- 改依赖：使用项目已有包管理方式，并说明依赖风险；
- 改站点配置：确认 Jekyll 配置字段没有破坏现有页面。

当前仓库的前端构建链较旧，普通 `npm install` 可能因 peer dependency 冲突失败；如需验证 Grunt，可使用兼容旧依赖的安装方式后再执行现有 Grunt 任务。

## 4. 文章发布规范

### 4.1 文件路径

新文章放在 `_posts/<category>/` 下，例如：

- `_posts/tools/`：工具、Agent、效率、Markdown、Git；
- `_posts/devops/`：部署、数据库、中间件、监控；
- `_posts/build/`：编译、构建系统、工具链；
- `_posts/debug/`：故障排查、调试、性能定位；
- `_posts/notes/`：方法论、读书笔记、复盘。

文件名使用：

```text
YYYY-MM-DD-topic.md
```

中文标题可以出现在文件名中，但优先保持简短、可读、稳定。

### 4.2 Front Matter

新文章保持与本站现有文章一致：

```yaml
---
layout:     post
title:      标题
subtitle:   副标题
date:       YYYY-MM-DD
author:     BY
header-img: img/post-bg-ios9-web.jpg
catalog: true
tags:
    - Tag
---
```

标签数量不宜过多，优先复用已有标签；新增标签应简洁、可长期使用。

### 4.3 内容结构

推荐结构：

1. 开头引用块：说明本文来源、整理目标或适用场景；
2. 背景：为什么需要这篇文章；
3. 核心结论：先给读者最重要的信息；
4. 操作步骤或方法论：按执行顺序展开；
5. 风险与排查：记录易错点；
6. Checklist / FAQ / 后续计划：方便复用。

## 5. Agent 协作原则

### 5.1 上下文管理

- 长任务要把关键结论落到文档中，不能只依赖会话历史；
- 如果切换模型或上下文被压缩，先回读相关文件再继续；
- 对用户已有表述保持尊重，优先整理和提炼，不擅自改写成完全不同的主题；
- 不确定时提出问题，避免自作主张。

### 5.2 工具使用

- 搜索内容优先用仓库内已有文件；
- 修改前确认工作区状态；
- 不提交 `node_modules`、临时文件、密钥或本地缓存；
- 不为一次性验证创建会污染仓库的脚本；
- 需要临时实验时使用 `/tmp`。

### 5.3 安全与质量

- 不把 token、cookie、私钥、账号密码写进博客；
- 不把未经脱敏的内部域名、IP、工单、客户信息写进博客；
- Shell 命令要避免危险删除、越权操作和不可逆操作；
- 示例代码优先安全实现，避免直接推荐 `eval`、明文凭据、关闭校验等做法；
- 引入外部依赖前先确认必要性和风险。

## 6. 完成标准

一次 blog 变更完成前，应满足：

- 文件路径和命名符合 `_posts/<category>/YYYY-MM-DD-title.md`；
- Front Matter 可被 Jekyll 解析；
- 标题、章节、标签与现有风格一致；
- 文章能独立阅读，不依赖读者回看原始对话；
- 命令、配置、日志格式清晰；
- 不包含敏感信息；
- 已按变更范围完成验证，并记录无法验证的原因。

如果文章来自 `_patch/`，还需要确认：

- 原始资料已经完成合并、拆分、脱敏和校对；
- 已从暂存路径移动到正确的 `_posts/<category>/` 路径；
- `_patch/` 中不再保留已经发布但无后续价值的重复稿。

## 7. 可复用提示词

后续维护本站时，可以直接引用：

```text
请按照《Blog Agent Skill 文档》的流程处理本次博客任务：
1. 先判断是解释、计划、内容还是工程类任务；
2. 修改前阅读相关目录和已有文章；
3. 新增或调整文章时保持 Front Matter、分类、标签和结构一致；
4. 批量专题先放入 _patch 暂存，整理达标后再移动到 _posts 发布；
5. 不提交临时文件、依赖目录或敏感信息；
6. 修改后按影响范围验证。
```

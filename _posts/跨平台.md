
1. 推荐采用 LF（\n） 作为 EOL
原因：
Linux/macOS 默认使用 LF（\n）。
Windows 默认使用 CRLF（\r\n），但现代编辑器（如 VSCode、CLion、Sublime、Notepad++）都能很好地识别 LF。
GitHub、CI/CD、跨平台工具链都推荐 LF，减少因 EOL 不一致导致的编译/补丁/合并冲突等问题。
2. Git 层面自动处理
.gitattributes 文件配置
在项目根目录添加或修改 .gitattributes 文件，加入如下内容：

Code

```
*.cpp text eol=lf
*.hpp text eol=lf
*.h   text eol=lf
*.c   text eol=lf
*.inl text eol=lf
*.cmake text eol=lf
```

这样，Git 在提交时会自动把这些文件的 EOL 转换为 LF，拉取时也会保持一致。
3. 编辑器设置
VSCode：可设置 "files.eol": "\n"，并用右下角的 EOL 按钮批量转换。
CLion/IDEA：File → Line separators → Unix and OS X (\n)
Notepad++：编辑 → EOL转换 → 转为 UNIX 格式
4. CMake/工具链层面
一般不需要特殊设置。但可以用代码格式化工具（如 clang-format）统一行尾。
如果有自动代码生成步骤，建议生成脚本强制用 LF（如 Python/Perl 脚本用 open(..., newline='\n')）。
5. 总结
最佳实践：用 .gitattributes 管控，开发工具用 LF，团队达成共识即可。
不建议：用 CRLF，除非你的目标平台仅限 Windows 并且有特殊历史兼容需求。
